# 기본 보고서 양식: 4열 테이블

<!-- This file defines the default Korean report format. Keep content in Korean — it is the output template. -->

월간 피드백 보고서의 기본 양식. 사용자가 별도 양식을 제공하지 않으면 이 형식을 사용한다.

## 양식

```markdown
# YYYY년 MM월 업무 피드백

| 목표 | 핵심결과 | 잘한 점 | 부족한 점 및 보완계획 |
|------|---------|--------|-------------------|
| [서비스명] 상위 목표 요약 | • 구체적 구현 내용 1 • 구체적 구현 내용 2 | • (카테고리) 성과 설명 | • 부족한 점 설명 • 보완계획: 구체적 액션 |
```

## 각 열 작성 가이드

**독자:** 기획자, 팀장 등 비개발자도 읽는 문서이며, **사내 평가 자료로 활용**될 수 있다. 개발 용어를 쓸 수 있지만, 모든 항목에서 **"이 작업이 서비스/제품/팀에 어떤 변화를 만들었는가"**를 중심으로 서술한다. 글머리 기호 나열만으로 끝내지 않고, 맥락과 임팩트를 설명하는 문장을 포함한다.

**핵심 원칙 — 임팩트 중심 서술:**
- 단순히 "무엇을 했다"가 아니라 **"그래서 서비스/팀에 어떤 좋은 영향을 주었는가"**를 반드시 포함
- 서비스 개선, 사용자 경험 향상, 팀 생산성 기여, 기술 부채 해소, 장애 예방 등 **비즈니스/조직 관점의 가치**로 연결
- 혁신이나 변화가 있었다면 (새로운 도구 도입, 프로세스 개선, 자동화 등) 구체적으로 before/after를 대비하여 강조
- 항목당 최소 2~3문장, 서술형(내러티브)으로 작성
- **단, 과장하지 않는다.** 실제로 일어난 사실과 데이터(커밋 수, 기간, 테스트 결과 등)에 근거하여 작성한다. "획기적", "혁신적" 같은 과한 수식어 대신, 구체적인 숫자와 before/after 비교로 임팩트를 보여준다. 추측이나 기대 효과는 "~할 것으로 기대됨" 등으로 명확히 구분한다.
- **문체:** 명사형 종결을 사용한다. "~했다", "~이다" 대신 "~했음", "~함", "~완성", "~개선", "~예정" 등으로 마무리한다.

### 목표
- 서비스명을 대괄호로 표시: `[서비스명]`
- 개별 커밋이 아닌 상위 목적을 작성
- 비즈니스 관점의 목표를 포함 (단순 기술 작업이 아닌, 이 작업이 왜 필요했는지)
- 예: "사용자 인증 플로우 개선 및 UI 고도화 — 기존 로그인 이탈률을 줄이고 소셜 로그인으로 가입 허들을 낮추기 위한 작업"

### 핵심결과
- 실제로 구현/완료한 것을 구체적으로 나열하되, 각 항목에 "무엇을 왜 했는지" 한 줄 설명을 덧붙인다
- 기술적 키워드를 포함하되, 그 기술이 사용자/제품에 주는 효과도 함께 기술
- 글머리 기호(•)로 구분, 항목당 1~2문장
- 예:
  - "• 로그인 페이지 리뉴얼 및 소셜 로그인(카카오) 연동 구현 — 기존 이메일/비밀번호만 지원하던 로그인에 카카오 원클릭 로그인을 추가하여 가입 전환율 개선 기대"
  - "• 부고 등록 폼 전체 구현 (에디터, 날짜 입력, 영정사진 업로드 포함) — 사용자가 웹에서 직접 부고를 작성하고 등록할 수 있는 핵심 플로우 완성"

### 잘한 점
- 괄호로 카테고리를 표시: `(생산성 향상)`, `(전반적인 품질 향상)`, `(디테일 챙기기)`, `(선제적 대응)`, `(협업)`
- 단순히 "잘했다"가 아니라 **구체적인 상황 → 판단 → 결과** 흐름으로 서술
- 항목당 2~3문장으로, 어떤 상황에서 어떤 선택을 했고 그 결과가 어땠는지 설명
- 예:
  - "• (생산성 향상) lifecanvas 클론 프로젝트에서 AI 에이전트를 활용해 15개 태스크를 병렬로 실행했다. 일반적으로 1~2주 걸릴 수 있는 프로젝트 초기 세팅과 핵심 기능 구현을 하루 만에 완료하고, 59개 테스트를 모두 통과시켜 품질까지 확보했다."
  - "• (선제적 대응) 커리큘럼 기능 개발 시 N+1 쿼리 문제를 사전에 발견하고 LEFT JOIN으로 전환했다. 데이터가 늘어나도 성능 저하 없이 안정적으로 동작하는 구조를 확보했다."

### 부족한 점 및 보완계획
- 부족한 점과 보완계획을 함께 작성
- **왜 부족했는지 배경**을 먼저 설명하고, **구체적인 다음 액션**(누가, 언제, 무엇을)으로 마무리
- 항목당 2~3문장으로, 단순 나열이 아닌 개선 방향의 맥락을 포함
- 예:
  - "• 부고/추모관 페이지의 API 실제 연동이 아직 완료되지 않았다. 백엔드 API 개발이 진행 중이어서 프론트에서는 목업 데이터로 먼저 UI를 완성한 상태이다. 백엔드 API가 확정되는 시점에 맞춰 연동 작업을 진행하고, 통합 테스트로 데이터 흐름을 검증할 예정이다."
  - "• 스킬 문서(SKILL.md)에서 스텝 번호 불일치 같은 사소한 오류가 반복되었다. 문서 구조가 복잡해지면서 수정 시 전체 흐름을 놓치는 경우가 생겼다. 문서 변경 시 체크리스트를 만들어 전체 흐름을 한번에 검증하는 프로세스를 추가할 예정이다."

## 서비스별 행 구분

서비스가 여러 개면 행을 나눈다:

```markdown
| 목표 | 핵심결과 | 잘한 점 | 부족한 점 및 보완계획 |
|------|---------|--------|-------------------|
| **[서비스A]** 사용자 인증 플로우 개선 | • 로그인 페이지 리뉴얼 ... | • (생산성 향상) ... | • 테스트 수동 검수 ... |
| **[서비스B]** 검색 기능 고도화 | • 필터링 UI 개선 ... | • (품질 향상) ... | • 테스트 강화 필요 ... |
```

## 업무 리스트 양식

업무 리스트는 별도 문서로 생성한다:

```markdown
# YYYY년 MM월 업무 리스트

## [서비스명] 레포명 (N일 작업, M커밋)

1. ✅ 완료된 태스크 제목
   - 기간: M/D ~ M/D (N일)
   - 커밋 N건, MR !번호 머지
   - 주요 결정: 간단한 설명

2. 🔧 진행 중인 태스크 제목
   - 기간: M/D ~ (진행중)
   - 커밋 N건

## 코드 외 업무
- 코드리뷰: N건 (서비스별 분류)
- 미팅: 목록
- 기타: ...

---
> 총 근무: N일, 평균 N시간/일, 총 N커밋
```
